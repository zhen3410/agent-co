# 对等讨论模式设计

**日期：** 2026-04-04
**状态：** 已通过会话讨论确认，待实现
**适用范围：** 聊天室多智能体会话

---

## 1. 目标

当前聊天室的链式调用机制更偏向“任务分发”：用户触发一批 agent，agent 通过 `invokeAgents` 或 `@@AgentName` 继续点名下一个 agent。这个机制适合有明确主控/执行关系的协作，但不适合真正对等的多方讨论。

本设计引入 **对等讨论模式（peer discussion mode）**，目标是：

- 不设固定主控 agent；
- 每个 agent 都可以自由提出质疑、反驳、追问与补充；
- 显式点名仍然是主要传播方式；
- 在没有显式点名时，系统允许有限度的“条件性主动插话”；
- 讨论在没有新有效推进时自然暂停；
- 暂停后不自动总结，由用户手动触发总结。

这个设计要与现有 classic 链式模式兼容，不破坏已有分发/串联能力。

---

## 2. 非目标

本次设计不解决以下问题：

- 不做复杂的全自动辩论裁判器；
- 不追求首版就实现高精度语义冲突检测；
- 不强制某个角色承担主持/收束责任；
- 不在首版引入复杂的多轮观点图谱或论证树；
- 不要求所有 agent 都必须主动发言。

---

## 3. 模式划分

建议为 session 增加讨论模式配置：

- `classic`：保持当前链式调用语义；
- `peer`：启用对等讨论模式。

推荐策略：

- 默认仍使用 `classic`，避免影响现有行为；
- 用户在需要“平等辩论/讨论”时切换到 `peer`；
- 后续可以在 UI 上给出模式说明，帮助用户理解二者差异。

---

## 4. 核心调度规则

### 4.1 初始触发

用户消息中的：

- `@某 agent`
- `@所有人`

仍然作为第一轮参与者集合。

### 4.2 显式继续

在 `peer` 模式下，agent 消息若包含以下任一继续信号，则进入下一轮：

- callback `invokeAgents`
- 文本中的 `@@AgentName`

这是讨论传播的主路径，优先级最高。

### 4.3 条件性主动插话

若一条 agent 消息未显式点名任何其他 agent，系统不应立刻认定讨论结束，而应进入一次“主动插话判定”。

若存在满足条件的 agent，则将其作为下一轮候选入队；若不存在，则讨论暂停。

### 4.4 不设固定主控

系统不应将最后一轮自动回交给某个固定角色（例如 CPO）完成收束。任何收束动作都应从“讨论流程”中解耦。

---

## 5. 主动插话判定

主动插话只在以下前提下发生：

1. 当前模式为 `peer`；
2. 当前消息没有显式继续对象；
3. 候选 agent 未超过本轮发言限制；
4. 候选 agent 的插话理由属于允许类别。

### 5.1 允许的插话理由

建议首版定义 4 类理由：

1. `factual_conflict`：事实纠错
2. `logical_conflict`：逻辑冲突
3. `role_boundary`：职责越界
4. `strong_relevant_objection`：强相关异议

### 5.2 不允许的插话情况

以下情况应被压制：

- 只是重复已有观点；
- 只是礼貌性附和；
- 与自身职责边界明显不符；
- 最近几轮已连续发言过多；
- 预计只会造成来回空转。

### 5.3 首版实现建议

首版不建议完全依赖后端做复杂语义推断，而应采用“**显式点名为主 + 保守型主动插话**”策略：

- 系统先保留现有显式链路机制；
- 只在少数高置信场景开放主动插话；
- 允许后续逐步增强判定器。

---

## 6. 停止条件

在 `peer` 模式下，讨论的停止条件应从“角色驱动”改为“讨论状态驱动”。

### 6.1 自然暂停条件

满足以下条件时，讨论暂停：

1. 当前消息没有显式继续对象；
2. 主动插话判定未找到合格候选者。

### 6.2 硬停止条件

建议保留或新增以下限制：

- 最大传播轮数（复用 `agentChainMaxHops` 语义）；
- 单 agent 发言上限；
- 后续可扩展：互怼对数上限、重复度过高停止等。

### 6.3 停止后的产品语义

讨论暂停不应被展示为失败，而应明确表达为：

- 当前没有新的有效回应；
- 用户可以继续点名某个 agent；
- 或用户手动触发总结。

---

## 7. 总结阶段

用户已明确选择：**讨论暂停后，由用户决定是否生成总结（手动总结）**。

因此：

- 讨论暂停后不自动总结；
- 不指定固定 agent 自动收尾；
- 前端提供 `生成总结` 操作入口；
- 总结动作与讨论传播解耦。

后续可选择：

- 由系统默认总结器处理；
- 由用户手动指定某个 agent 总结；
- 提供“中立总结 / 观点对照 / 行动项提炼”等不同总结类型。

首版只需确保“生成总结”是一个独立动作即可。

---

## 8. 前端交互设计

### 8.1 讨论中

建议在内部或调试视图中保留每条消息的传播来源信息，例如：

- 显式点名继续
- 主动插话
- 质疑 / 反驳 / 补充

首版不要求全部展示给普通用户，但建议保留元数据，便于调试与后续演进。

### 8.2 讨论暂停卡片

当讨论自然暂停时，在消息流底部展示状态卡：

- 标题：`讨论已暂停`
- 说明：`当前没有新的有效回应。你可以继续点名某个智能体，或生成总结。`

建议按钮：

- `生成总结`
- `继续讨论`

### 8.3 继续讨论

用户可通过两种方式恢复：

- 直接输入 `@某 agent` 继续点名；
- 点击 UI 提供的“继续讨论”入口并选择目标 agent。

### 8.4 总结入口

`生成总结` 应视为一个新动作，而不是把总结混入继续讨论队列。

---

## 9. 数据结构建议

### 9.1 Session 级字段

建议增加：

- `discussionMode: 'classic' | 'peer'`
- `discussionState: 'active' | 'paused' | 'summarizing'`

### 9.2 更细粒度的 dispatchKind

当前为：

- `initial`
- `chained`

建议扩展为：

- `initial`
- `explicit_chained`
- `implicit_chained`
- `summary`

### 9.3 Message 元信息

建议消息结构可选增加：

- `invokeAgents`
- `discussionIntent`
  - `question`
  - `challenge`
  - `rebuttal`
  - `evidence`
  - `proposal`
- `dispatchKind`
- `triggerReason`
  - `explicit_mention`
  - `factual_conflict`
  - `logical_conflict`
  - `role_boundary`
  - `strong_relevant_objection`

首版这些字段均可设计为可选，以保证兼容。

---

## 10. 与现有系统的兼容策略

### 10.1 classic 不变

若 session 未启用 `peer` 模式，则沿用当前 classic 逻辑：

- 只依赖 `invokeAgents` / `@@AgentName` 继续；
- 不进行主动插话判断；
- 保持当前已有测试与使用习惯。

### 10.2 peer 作为增量能力

`peer` 模式只在显式启用时生效，避免一次性改变所有会话的链式行为。

### 10.3 总结动作独立

手动总结不应复用普通 chained dispatch 的所有语义，建议在后端拥有单独入口或单独调度类型。

---

## 11. 推荐落地顺序

### 第一阶段：最小闭环

目标：先把“对等讨论暂停 + 用户手动总结”的产品闭环打通。

范围：

- session 增加 `discussionMode`；
- `peer` 模式保留显式点名传播；
- 讨论无法继续时标记 `discussionState='paused'`；
- 前端展示“讨论已暂停”卡片；
- 提供 `生成总结` 按钮入口；
- 总结动作单独触发。

### 第二阶段：主动插话

目标：在不失控的前提下增加“有限自由讨论”。

范围：

- 引入 `implicit_chained`；
- 增加保守型主动插话判定；
- 增强日志与调试信息。

### 第三阶段：讨论质量优化

目标：提升讨论质量与稳定性。

范围：

- 重复观点抑制；
- 互怼上限；
- 更细粒度意图标签；
- 更智能的暂停与恢复体验。

---

## 12. 风险与权衡

### 风险 1：主动插话过多导致刷屏

缓解：

- 首版保守开放；
- 保留总轮数与单 agent 上限；
- 先不开全量广播。

### 风险 2：用户不理解“暂停”和“失败”的区别

缓解：

- 前端使用明确文案；
- 将暂停态设计成正常讨论结束，而不是报错。

### 风险 3：对等讨论失去收束能力

缓解：

- 将总结动作独立出来；
- 用户手动触发总结；
- 不把收束权固化到某个角色。

### 风险 4：与现有链式测试冲突

缓解：

- `peer` 模式做 session 级开关；
- classic 路径保持不变；
- 为 peer 模式新增独立测试集。

---

## 13. 结论

本设计的核心不是让“最后一次一定串回某个主控 agent”，而是把系统从“任务分发链”升级为“对等讨论流”：

- 传播由显式点名主导；
- 自由讨论通过有限主动插话补充；
- 终止条件改为状态驱动而非角色驱动；
- 总结与讨论解耦，并交由用户手动触发。

这能最大程度保留“多 agent 平等讨论、互相质疑”的产品特性，同时控制复杂度，保证与现有 classic 链式模式共存。
