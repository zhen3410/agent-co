# 前端 MPA React 重构设计

**日期：** 2026-04-12
**状态：** 已通过会话讨论确认，待细化实施计划
**适用范围：** 聊天主站、管理后台、监控/日志/辅助页面的前端整体重构

---

## 1. 目标

当前前端以裸写 HTML + 独立脚本为主，页面入口分散在 `public/` 与 `public-auth/` 下，存在以下问题：

- 页面结构、样式、脚本耦合在单文件或少量全局文件中；
- 页面之间缺乏统一的组件抽象、设计语言和状态模型；
- 聊天页、后台页、运维页的交互模式和布局策略各自为政；
- 后续新增功能时，容易继续堆积 DOM 操作与全局样式；
- 现有仓库后端已完成较清晰的 `chat / admin / shared` 分层，但前端尚未形成对应边界。

本次重构目标是建立一个可长期演进的前端应用层，以 **React + Vite + TypeScript + MPA** 为基础，对全站前端进行统一升级，同时兼顾工程可维护性与界面体验提升。

核心目标如下：

1. 建立独立、可维护的前端工程结构；
2. 用 MPA 替代当前零散的静态 HTML 页面组织方式；
3. 统一聊天、后台、监控/日志页面的布局系统、视觉基础与交互基建；
4. 抽离共享组件、共享 API 层、共享实时通信封装；
5. 在不推倒现有 Node/TypeScript 服务端分层的前提下，完成前后端职责重划。

---

## 2. 非目标

本次设计不包含以下内容：

- 不重写现有后端业务逻辑与会话/管理/运维接口；
- 不引入 BFF 或 SSR 体系，首版仍以静态资源分发 + API/WS 驱动渲染为主；
- 不在首版引入复杂微前端架构；
- 不在首版彻底重构所有后端返回结构，只在必要处增加前端 mapper / adapter；
- 不要求一次性移除所有旧页面，可接受分阶段迁移与短期双轨并存；
- 不把 mockup/临时演示页当成长期产品能力重点建设对象。

---

## 3. 用户确认的关键约束

本次会话中已确认以下前提：

### 3.1 范围

重构范围不是只覆盖聊天页，而是覆盖 **全站前端页面**，包括：

- 聊天主站；
- 管理后台；
- 监控/日志/辅助页面。

### 3.2 技术栈

前端主栈确认为：

- **React**
- **Vite**
- **TypeScript**

### 3.3 页面形态

采用 **MPA（多页面应用）**，而不是单一 SPA。

### 3.4 优先级

本次方案优先级为：

- **可维护性优先**；
- 同时兼顾视觉升级与交互重设计；
- 不采用“仅视觉翻新但保留裸写 HTML 架构”的路线。

---

## 4. 方案对比与结论

### 4.1 方案 A：完整前端子应用化（推荐）

做法：新增独立前端工程目录，使用 React + Vite + TypeScript 搭建 MPA，多入口分别承载 chat、admin、ops 页面，构建产物重新接回 Node 服务静态托管。

优点：

- 结构最清晰，长期维护成本最低；
- 可系统性统一组件、样式、API、实时通信模型；
- 便于按业务域拆分；
- 有利于测试、构建、页面复用和后续演进。

缺点：

- 初始迁移成本最高；
- 需要重新设计前端构建产物与现有服务端静态目录的衔接方式；
- 首批迁移需要明确页面优先级与双轨过渡策略。

### 4.2 方案 B：渐进式 React 岛屿化

做法：保留现有 HTML 页面壳子，在复杂区域内部挂载 React 组件，逐步替换页面局部。

优点：

- 风险较低；
- 局部可快速见效；
- 对现有静态托管模式影响较小。

缺点：

- 长期会形成 HTML 壳 + React 岛的双重复杂度；
- 页面边界、样式边界、状态边界容易继续混乱；
- 不适合“全站统一重构”。

### 4.3 方案 C：先做设计系统，后迁工程架构

做法：先抽象主题、样式 token、公共样式与少量 JS 模块，不立即迁移页面到 React/Vite。

优点：

- 上手快；
- 可先统一部分视觉语言。

缺点：

- 无法从根本上解决裸写 HTML 与 DOM 脚本堆积问题；
- 容易只停留在“有规范、无落地”的中间状态。

### 4.4 结论

采用 **方案 A：完整前端子应用化**。

原因：

- 用户已明确要做全站统一重构；
- 技术栈与页面形态已指向 React + Vite + TypeScript + MPA；
- 优先级是可维护性优先，而不是低风险局部翻修；
- 仓库后端已有明确业务域划分，前端可顺势建立对称结构。

## 5. 总体架构设计

### 5.1 前后端职责重划

重构后，系统职责建议如下：

- **Node 服务端继续负责：**
  - API 路由；
  - WebSocket / SSE / 回调等实时接口；
  - 身份校验与权限控制；
  - 静态资源托管；
  - 各业务域运行时与存储。

- **前端应用负责：**
  - 页面渲染；
  - 组件组合与交互逻辑；
  - 前端路由入口（页面级，而非 SPA 内路由）；
  - 服务端数据消费、缓存与订阅；
  - 设计系统、主题与用户体验一致性。

也就是说，后端不再承担前端页面结构层面的组织责任，前端成为正式的一等应用层。

### 5.2 新增独立前端目录

建议新增目录：`frontend/`。

推荐形态：

```text
frontend/
  index.html                # 可选：Vite 根入口占位
  vite.config.ts
  tsconfig.json
  src/
    entries/
      chat-main.tsx
      admin-main.tsx
      deps-monitor-main.tsx
      verbose-logs-main.tsx
      support-main.tsx      # 视最终页面而定
    chat/
      pages/
      features/
      components/
      hooks/
      services/
      types/
    admin/
      pages/
      features/
      components/
      hooks/
      services/
      types/
    ops/
      pages/
      features/
      components/
      hooks/
      services/
      types/
    shared/
      ui/
      layouts/
      hooks/
      lib/
      styles/
      types/
      config/
```

原则：

- **按业务域拆分，不按技术层平铺全站页面；**
- 每个域拥有自己的页面、业务组件、hooks、services；
- 全站共享能力集中在 `shared/`。

### 5.3 MPA 入口设计

每个产品页面保留独立入口，而不是做一个单壳 SPA。这样更符合当前站点结构，也更利于服务端按已有 URL 方式托管。

建议入口映射：

| 旧页面 | 新前端入口 | 目标域 |
|---|---|---|
| `public/index.html` | `entries/chat-main.tsx` | `chat` |
| `public-auth/admin.html` | `entries/admin-main.tsx` | `admin` |
| `public/deps-monitor.html` | `entries/deps-monitor-main.tsx` | `ops` |
| `public/verbose-logs.html` | `entries/verbose-logs-main.tsx` | `ops` |
| 其他辅助页 | 按需拆为独立入口 | `ops` / `shared` |

如仍需保留某些纯静态说明页，可允许极少数页面不迁移，但原则上产品页全部纳入统一前端工程。

## 6. 页面与模块分层

### 6.1 业务域划分

建议把全站前端拆成三个主要域：

#### Chat 域

覆盖聊天主站核心功能：

- 会话列表与切换；
- 消息流展示；
- 输入区与发送控制；
- 智能体状态与调用反馈；
- 时间线 / 调用链视图；
- 增量同步、断线重连、恢复执行等交互。

#### Admin 域

覆盖认证后管理后台：

- 用户与权限管理；
- 分组/agent 管理；
- 模型连接配置；
- 系统级配置与运维入口。

#### Ops 域

覆盖运维与诊断工具：

- 依赖监控；
- 详细日志；
- 健康状态；
- 诊断/调试辅助页。

### 6.2 页面内部结构

每个页面建议遵循统一层次：

1. **Page**：页面容器，负责参数解析、页面级装配、权限/初始化；
2. **Feature Section**：业务分区，如消息区、筛选区、配置区；
3. **Feature Component**：业务组件，如会话侧栏、日志 viewer、模型配置表；
4. **Shared UI Component**：按钮、表单、卡片、表格、标签、对话框等；
5. **Hook / Service / Mapper**：数据获取、订阅、状态拼装、数据转换。

目标是避免再次出现“一个页面 = 一个大 HTML + 一堆选择器操作”的结构。

## 7. 组件体系设计

### 7.1 通用 UI 层

建议在 `frontend/src/shared/ui/` 中建立基础组件层，包括但不限于：

- `Button`
- `IconButton`
- `Input`
- `Textarea`
- `Select`
- `Checkbox`
- `Switch`
- `Modal`
- `Drawer`
- `Tabs`
- `Badge`
- `Card`
- `Table`
- `EmptyState`
- `ErrorState`
- `Spinner`
- `Tooltip`
- `Toast`

这些组件承担统一视觉语言与交互反馈。

### 7.2 业务组件层

按域构建业务组件，例如：

- Chat：`SessionSidebar`、`ChatMessageList`、`ChatComposer`、`TimelinePanel`、`AgentStatusBadge`
- Admin：`AgentConfigTable`、`UserManagementPanel`、`ModelConnectionForm`
- Ops：`DependencyHealthPanel`、`VerboseLogViewer`、`LogFilterBar`

规则：

- 业务组件可以依赖通用 UI；
- 通用 UI 不依赖具体业务域；
- 跨域复用的组件优先上移到 `shared/`。

## 8. 数据、状态与实时通信

### 8.1 状态分层原则

不建议把所有前端状态都塞进单一全局 store，而应按职责分层：

#### A. 服务端状态

用于：

- 列表、详情、配置、统计等来自 HTTP 的数据；
- loading / error / stale / refresh 生命周期。

建议通过统一数据访问层与请求缓存机制管理。

#### B. 实时事件状态

用于：

- WebSocket / SSE 事件流；
- 断线重连；
- 增量事件归并；
- 聊天时间线与执行反馈。

建议抽象独立客户端封装，而非在组件中直接裸连。

#### C. 本地 UI 状态

用于：

- 面板开关；
- 当前 tab；
- 表格筛选器；
- 草稿内容；
- 暂时性的可视交互状态。

这部分应尽量局部化，减少无必要的全局共享。

### 8.2 API Client 设计

建议按业务域封装 API Client：

- `chatApi`
- `adminApi`
- `opsApi`

统一处理：

- base URL 与路径拼装；
- 公共 headers；
- 鉴权信息透传；
- 错误对象规范化；
- 响应数据解析；
- 超时与取消。

### 8.3 实时通信封装

聊天页与部分运维页有实时性需求，建议在 `shared/lib/realtime/` 建立统一封装，负责：

- 建连与断连；
- 自动重连；
- 订阅/解绑；
- 事件分类与反序列化；
- 前端 view model 更新入口。

重点是将“事件流处理”从页面组件中抽离，避免业务逻辑散落在 UI 层。

## 9. 视觉与设计系统

### 9.1 总体方向

产品同时包含聊天、后台和运维工具，设计风格应偏：

- 专业、克制；
- 信息层级清晰；
- 高密度但不压迫；
- 强状态反馈；
- 支持浅色/深色主题。

不建议走营销页式高装饰风格。

### 9.2 设计基础层

建议建立统一 design tokens：

- 颜色系统；
- 字体与字号层级；
- 间距体系；
- 圆角；
- 阴影；
- 边框/分隔线；
- 状态色；
- 焦点态/悬停态/禁用态。

这些 token 以 CSS Variables 为基础输出，供全站共享。

### 9.3 三类场景化模板

在统一设计系统下，允许不同域有轻量差异：

- **Chat：** 更强调沉浸式内容区、消息层级、协作感；
- **Admin：** 更强调配置效率、表格密度、信息组织；
- **Ops：** 更强调日志可读性、状态扫读能力、诊断效率。

这意味着统一基础组件 + 分域布局模板，而不是完全相同的页面皮肤。

## 10. 与现有仓库结构的衔接

### 10.1 与后端业务域对齐

现有后端已经按以下结构组织：

- `src/chat/`
- `src/admin/`
- `src/shared/`

前端重构应尽量对齐这一思路，避免出现新的命名体系与业务切分方式。

### 10.2 顶层 composition root 保持薄

现有规范强调：

- `src/server.ts`
- `src/auth-admin-server.ts`

保持轻量 composition root。

前端接入时也应遵守同样原则：

- 服务端路由层只负责静态资源与 API 装配；
- 不把前端构建逻辑和页面复杂性重新塞回顶层入口。

### 10.3 静态资源衔接策略

建议最终由 Vite 生成构建产物，再落到后端可托管目录中。可选方式：

1. 输出到新的构建目录，再由服务端显式挂载；
2. 构建后同步到现有 `public/` / `public-auth/` 对应目录；
3. 统一输出到单一前端 dist，再按路径映射给 chat/admin 服务使用。

首选是 **单独前端 dist + 服务端显式托管映射**，这样比直接把产物覆盖进现有手写静态目录更清晰。

## 12. 分阶段实施建议

### Phase 1：前端底座建设

目标：建立新前端工程，但不急于一次替换所有页面。

包括：

- 初始化 `frontend/`；
- 配置 Vite MPA；
- 建立共享样式 token 与基础 layout；
- 建立基础 UI 组件；
- 建立 API Client 与实时通信骨架；
- 完成构建产物与 Node 服务的托管联调。

### Phase 2：Chat 页面迁移

目标：先迁最核心、最复杂的聊天页。

包括：

- 拆分消息流、输入区、会话侧栏、时间线区域；
- 接入现有 HTTP / WS / SSE 能力；
- 统一加载态、错误态、空态；
- 保证主链路功能等价。

### Phase 3：Admin 页面迁移

目标：把后台配置页面纳入统一组件和表单体系。

包括：

- 登录后容器布局；
- 配置表格、编辑表单、状态反馈；
- 与后端 admin API 对接。

### Phase 4：Ops / Tooling 页面迁移

目标：统一监控与日志页面体验。

包括：

- 日志 viewer；
- 依赖状态面板；
- 筛选器与高密度表格；
- 统一工具页导航与布局。

### Phase 5：治理与优化

包括：

- 主题切换；
- 响应式与移动端策略；
- 可访问性；
- 单元测试/集成测试补齐；
- 性能优化与 bundle 治理；
- 移除旧 HTML 与遗留脚本。

---

## 13. 风险与控制点

### 13.1 范围过大

全站重构天然范围较大，应通过分阶段落地控制风险，避免一次性大爆炸替换。

### 13.2 旧接口不适合前端组件化消费

部分接口可能过于面向历史页面逻辑，需要通过 mapper 适配；必要时再小幅调整后端返回结构。

### 13.3 聊天实时链路复杂

聊天页涉及消息流、增量同步、断线恢复、执行状态等，必须优先定义清楚状态边界，否则迁移时最容易失控。

### 13.4 工具页被边缘化

若只重点设计 chat/admin，ops 页面容易退回“随手写”，因此必须从第一阶段就纳入 shared layout 与 shared ui 体系。

---

## 14. 验收标准

当以下条件满足时，可认为本轮重构达成目标：

1. 全站主要产品页均已迁入 `frontend/` 管理；
2. 旧的核心裸写 HTML 页面不再承担正式产品主入口；
3. Chat/Admin/Ops 三域共享统一设计 token 与基础组件；
4. HTTP 请求与实时连接不再在页面组件中散落裸写；
5. 静态资源构建与服务端托管路径稳定；
6. 新增前端页面与功能可以按既定目录和抽象继续扩展，而非重新回到页面脚本模式。

---

## 15. 结论

本次前端重构建议采用：

- **React + Vite + TypeScript**
- **MPA 多入口模式**
- **独立 `frontend/` 工程**
- **按 chat / admin / ops 分域组织**
- **共享 design system + API/实时通信基建**
- **分阶段迁移现有 `public/` 与 `public-auth/` 页面**

该方案既能解决当前“裸写 HTML”带来的结构性问题，也能为后续聊天体验升级、后台配置扩展和运维工具统一提供稳定基础。
