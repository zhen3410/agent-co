# 前端整体重构设计

**日期：** 2026-04-13  
**状态：** 已通过会话讨论确认，待用户审阅与实施计划拆解  
**适用范围：** 首页/入口页、登录/鉴权页、聊天主界面、管理端，以及对应桌面端与移动端体验

---

## 1. 背景与问题

仓库已在 `frontend/` 下完成 React + Vite + TypeScript + MPA 基础重构，当前主要页面入口和共享样式已具备统一承载能力：

- 页面入口：`frontend/src/entries/chat-main.tsx`、`admin-main.tsx` 等；
- 聊天主页面：`frontend/src/chat/pages/ChatPage.tsx`；
- 管理端页面：`frontend/src/admin/pages/AdminPage.tsx`；
- 共享样式基础：`frontend/src/shared/styles/tokens.css`、`base.css`；
- 共享 UI 基础：`frontend/src/shared/ui/*`、`frontend/src/shared/layouts/*`。

但从当前产品观感与使用体验来看，前端仍处于“工程骨架已搭起、产品界面语言尚未统一”的阶段，存在以下系统性问题：

1. **整体视觉语言不足统一**：布局、颜色、排版、交互反馈之间未形成稳定设计系统；
2. **聊天页缺少明确主次关系**：对话区、输入区、侧栏、状态区彼此争抢注意力；
3. **组件与容器存在模板化/后台化倾向**：容易落入厚卡片、宫格式分块、泛 SaaS/AI 产品视觉套路；
4. **首页、登录页、聊天页、管理端缺少“同一产品”的连续感**；
5. **移动端策略尚未成为一等设计目标**：容易退化为桌面布局压缩版，而不是独立成立的移动体验。

这次重构不是一次简单“换皮”，而是一次 **结构级前端体验重构**：在保持现有业务功能边界的前提下，重做整套设计系统、页面骨架、层级关系与跨端策略。

---

## 2. 目标

### 2.1 总目标

为当前项目建立一套面向 **个人开发者与小团队** 的统一前端设计系统，使产品形成以下整体感受：

- 干净
- 柔和
- 现代
- 精致
- 内容感强
- 非“卡片 AI 风”

### 2.2 体验目标

本次重构需要同时达成：

1. **统一品牌气质**：让首页、登录页、聊天页、管理端属于同一产品体系；
2. **强化内容优先**：让消息、配置、状态、说明而非装饰性容器成为界面主角；
3. **提升聊天核心体验**：对话阅读与输入效率优先；
4. **建立双主题体系**：支持白天/夜间场景自然切换；
5. **补齐移动端策略**：移动端具备原生 app 感，而不是桌面缩放版；
6. **减少旧式后台感与模板化 AI 感**：避免传统 admin 宫格卡片与 AI SaaS 常见霓虹/悬浮套路。

---

## 3. 非目标

本次设计不包含以下内容：

- 不重写后端业务接口与 chat/admin/ops 分层；
- 不新增新的大型业务模块；
- 不改变 React + Vite + TypeScript + MPA 的既定工程路线；
- 不追求“首页营销化品牌再造”；
- 不在首轮就做过度拟物、插画驱动或强品牌图形系统；
- 不要求移动端完全等价承载桌面端全部复杂管理能力。

---

## 4. 用户已确认的关键约束

本次会话已确认以下设计前提：

### 4.1 范围

重构范围覆盖整个前端体系，而不是只改聊天页：

- 聊天主界面；
- 首页/入口页；
- 登录/鉴权页；
- 管理端。

### 4.2 目标用户

主要面向：

- **个人开发者**；
- **小团队**。

### 4.3 核心气质

用户明确要求整体风格为：

- 精致；
- 柔和；
- 现代；
- 内容感强；
- 避免“卡片 AI 风”。

### 4.4 品牌倾向

产品整体倾向 **创作者型**，但仍要保留工程理性与开发者工具属性。

### 4.5 主题要求

产品必须支持 **深浅双主题切换**，且两种主题都要独立成立，适配白天和晚上的使用场景。

### 4.6 首页/登录页的优先表达

表达顺序为：

1. 这是一个认真打磨过的开发者工具；
2. 它适合小团队协作；
3. 用户可以很快开始使用。

### 4.7 聊天页优先级

聊天页必须 **对话优先**，消息阅读与输入体验优先于侧栏和辅助状态展示。

### 4.8 管理端方向

管理端应与聊天端保持 **高度统一的设计语言**，只是信息密度略高，而不是另一套传统后台皮肤。

### 4.9 AI / agent 表达强度

允许适度体现 agent / workflow / 命令行语境，但不能走浮夸 AI 视觉路线。

### 4.10 当前最大问题

用户对当前前端的整体问题评价为：

- 布局不行；
- 颜色不行；
- 交互不行；
- 排版不行。

### 4.11 移动端定位

移动端聊天页明确采用：

- **原生 app 感**；
- **顶部导航 + 侧滑会话抽屉**。

## 5. 方案比较与结论

### 5.1 方案 A：温和重构

做法：保留现有页面骨架，只升级配色、字体、按钮、间距和局部组件皮肤。

优点：

- 风险较低；
- 上线速度快；
- 对现有 React 结构影响小。

缺点：

- 只能改善“表面观感”；
- 无法根治聊天页层级不清、管理端后台感重、首页入口模板化等核心问题；
- 容易出现“比现在好一些，但仍不像一个新产品”的结果。

### 5.2 方案 B：结构级重构（推荐）

做法：保留现有业务功能边界与页面边界，但重做设计系统、页面骨架、信息层级、跨端布局和组件组织规则。

优点：

- 能同时解决布局、颜色、交互、排版的系统性问题；
- 能让首页 / 登录 / 聊天 / 管理端形成统一产品语言；
- 更适合当前已经具备 React MPA 工程承载能力的前端基础。

缺点：

- 工作量中等；
- 需要先把统一设计系统与页面结构想清楚，再分阶段落地。

### 5.3 方案 C：品牌级重塑

做法：连带品牌表达、视觉符号、页面叙事、产品文案一起全面重做。

优点：

- 辨识度最高；
- 对外形象变化最明显。

缺点：

- 范围过大，超出当前重构“产品体验与信息结构优化”的必要边界；
- 容易稀释对核心聊天/管理工作流的关注。

### 5.4 结论

采用 **方案 B：结构级重构**。

原因：

- 用户明确对当前前端的整体布局、颜色、交互、排版均不满意；
- 问题已超出单纯视觉换肤范畴；
- 当前工程结构已具备系统性重构的承载条件；
- 相比品牌级重塑，结构级重构能更聚焦产品核心工作流。

## 6. 总体设计定位

### 6.1 产品定位语句

本次前端重构的最终定位语句为：

> **面向开发者与小团队的内容优先型工作台。**

### 6.2 设计关键词

统一关键词如下：

- 精致
- 柔和
- 现代
- 内容优先
- 工程理性
- 协作友好
- 非模板化 AI 产品

### 6.3 明确要避免的方向

本次重构需要主动规避：

1. **卡片 AI 风**：大面积独立悬浮卡片、发光边框、蓝紫霓虹渐变；
2. **传统后台风**：灰底 + 白卡片宫格 + 厚重表格；
3. **网页缩放式移动端**：多栏布局硬压缩到手机；
4. **终端皮肤化**：把整个产品做成 CLI 视觉装扮；
5. **营销模板风**：首页全部依赖 feature card 栅格与大块口号堆叠。

## 7. 设计系统总纲

### 7.1 核心原则

#### 原则 1：弱容器，强秩序

界面组织主要依赖：

- 间距；
- 字体层级；
- 对齐；
- 分组关系；
- 轻边框；
- 轻底色。

不依赖厚卡片和重阴影来制造层次。

#### 原则 2：内容优先

消息、配置项、说明文本、状态信息是界面主角，视觉容器只承担组织作用，不喧宾夺主。

#### 原则 3：双主题并重

- 浅色主题：轻、柔、适合白天长时间阅读；
- 深色主题：沉静、聚焦、适合夜间与工程语境。

两者都应独立成立，不做机械反相。

#### 原则 4：保留开发者语境，但不过度极客化

可通过命令、模型、工作流、状态语言表达产品属性，但不以 CLI 拟态统治整个界面。

## 8. 色彩系统

### 8.1 色彩结构

采用 **中性色为主 + 一支冷色主色 + 一支低饱和成功色** 的体系。

#### 中性色

中性色承担主要视觉面积，应占整体视觉的 80% 以上，用于背景、版面层次、文字、边框、分割。

#### 主色

主色建议落在 **slate-indigo / muted blue** 区间：

- 用于主按钮、链接、选中、焦点、关键动作；
- 避免高饱和科技蓝或蓝紫荧光色。

#### 成功色

成功色建议落在 **sage / muted emerald** 区间：

- 用于成功状态、运行中、可用性提示；
- 不与主 CTA 抢主角色。

#### 警示 / 错误色

- 警示色：amber / clay；
- 错误色：偏深 rose / red；
- 保持克制，不用刺眼高纯度色块。

### 8.2 浅色主题

浅色主题建议：

- `bg.canvas`：暖灰/石板感浅底，不使用纯白；
- `bg.surface`：内容表面，比画布略亮；
- `bg.subtle`：极轻分组底色；
- `border.default`：轻但明确的分割线；
- `text.primary`：深 slate；
- `text.secondary`：中深 slate；
- `text.tertiary`：用于说明与元信息。

### 8.3 深色主题

深色主题建议：

- `bg.canvas`：graphite / dark slate，而非纯黑；
- `bg.surface`：比背景略亮一层；
- `bg.elevated`：弹层、输入区、浮层；
- `border.default`：低对比但可见；
- `text.primary`：高亮浅灰白；
- `text.secondary`：柔和次级浅灰；
- `text.tertiary`：弱提示信息。

### 8.4 色彩使用规则

- 大面积背景不使用品牌色；
- 主色只服务于重点交互与焦点引导；
- 层次优先通过中性色明度差建立；
- 状态色优先用于语义表达，不用于装饰。

## 9. 字体与排版系统

### 9.1 字体策略

建议采用：

- **主 UI 字体**：Inter / IBM Plex Sans / Public Sans 路线；
- **Mono 字体**：JetBrains Mono / IBM Plex Mono；
- **首页增强气质**：仅在首页和个别大标题处使用轻微 editorial 处理，可通过字重、字距、排版节奏完成，不强依赖全站换成高个性标题字体。

### 9.2 字号层级

建议固定 type scale：

- 12：辅助说明 / 标签；
- 13-14：提示文案 / 元信息；
- 15-16：正文基础；
- 18-20：区块标题；
- 24-28：页面标题；
- 36-48：首页主标题。

### 9.3 排版原则

- 标题与正文关系稳定，不允许随页面自由漂移；
- 正文宽度控制，避免长行阅读压力；
- 元信息弱化但可见，不与主内容抢节奏；
- 聊天消息需首先满足“可读文本块”的阅读体验。

---

## 10. 间距、圆角、边框与阴影

### 10.1 间距系统

建立统一 spacing scale，例如：

- 4 / 8 / 12 / 16 / 24 / 32 / 48 / 64

并要求：

- 组件内部间距固定；
- 区块间距固定；
- 标题到正文、正文到操作区的节奏固定。

### 10.2 圆角

- 以小到中圆角为主；
- 避免过大圆角带来的玩具感与模板感。

### 10.3 边框

- 边框比阴影更重要；
- 通过轻边框和分割线组织层次，而非大面积阴影。

### 10.4 阴影

- 阴影只作为辅助；
- 弹层、浮层、个别 elevated panel 才可使用轻阴影；
- 不通过重阴影制造“悬浮式产品感”。

---

## 11. 组件系统规则

### 11.1 Button

#### Primary Button

- 实心；
- 对比明确；
- 圆角中等；
- hover 只做明度/底色变化；
- 不做 scale 或跳动。

#### Secondary Button

- 轻底色或轻边框；
- 用于次主流程操作。

#### Ghost / Text Button

- 用于工具栏、列表项操作、次级入口。

### 11.2 Input / Textarea

输入控件应更像工作台输入器，而不是传统表单框：

- 边界清晰；
- 与背景有轻层差；
- focus ring 克制但明确；
- 长文本输入时版面稳定；
- 附件、命令、模型选择器与输入区属于同一家族。

### 11.3 Tab / Segmented Control

- 视觉应偏轻、偏线性；
- 不建议使用厚重 pill 群组；
- 更适合管理端与工具页控制区。

### 11.4 Tag / Badge / Status Pill

- 优先浅底 + 文字 + 细边框；
- 少用高饱和实心 badge；
- 成功 / 运行中 / 失败需统一语义规则。

### 11.5 Panel / Section

默认组织方式应优先是：

- section 标题；
- 说明文案；
- 内容块；
- 分割线。

只有在确实需要强化边界时才使用 elevated panel，而不是所有内容都塞进 card。

### 11.6 Modal / Drawer / Sheet

- 深浅主题下都要稳；
- 不做沉重遮罩 + 大白盒式后台弹窗；
- 更偏内容导向和轻量控制台语义。

## 12. 页面级设计

### 12.1 首页 / 入口页

#### 页面目标

首页要优先建立三层认知：

1. 这是一个认真打磨过的开发者工具；
2. 它适合小团队协作；
3. 用户可以快速开始使用。

#### 页面结构

建议采用 **内容驱动型入口页**，而不是泛 SaaS 营销站：

1. **Hero 区**：
   - 左侧是简洁价值主张；
   - 右侧是产品真实界面片段或工作流片段；
   - 主 CTA + 次 CTA 即可。

2. **能力说明区**：
   - 采用纵向叙事式内容；
   - 每段配极简界面片段或结构图；
   - 不使用多张 feature card 横排作为主结构。

3. **信任与细节区**：
   - 强调适合的人群、工作方式、协作特征、模型/状态/工作流能力。

#### 设计定位

首页更像 **产品编辑页**，而不是广告页。

### 12.2 登录 / 鉴权页

#### 页面目标

传递“进入工作台”的感觉，而不是传统居中登录框。

#### 页面结构

- 上部：产品名 + 一句短价值表达；
- 中部：登录表单；
- 下部：辅助说明 / 错误提示 / 次级入口。

#### 设计规则

- 不做厚重悬浮卡片；
- 背景允许存在极弱结构，如柔和网格、模糊面、轻渐层；
- 输入框、按钮、错误态与加载态需统一且专业；
- 宽度控制在舒适阅读尺度内。

### 12.3 聊天主界面

#### 页面目标

构建 **对话优先** 的主工作界面。

#### 桌面端结构

采用 **单主舞台 + 收敛式辅助结构**：

- 主区：消息流；
- 底部：composer 输入区；
- 左侧：弱化的会话导航；
- 次级信息：模型、agent、执行状态、摘要等放入不打扰主区的次级层。

#### 重点问题与对应策略

1. **消息区不够像阅读区** → 控制行宽、段落节奏、消息分层；
2. **输入区不够像创作区** → 稳定、柔和、可扩展的 composer；
3. **侧栏和状态区抢主线** → 默认弱化，只在需要时增强；
4. **界面像拼装而非整体工作台** → 统一排版、色彩、边界和反馈语言。

#### 消息规则

- 用户消息：可有轻背景，但避免娱乐化气泡；
- 智能体消息：更偏正文阅读块；
- 系统/状态消息：更像时间线或运行记录，不与主回答抢视觉权重。

### 12.4 管理端

#### 页面目标

管理端是同一产品里的 **高密度控制台模式**。

#### 设计规则

- 沿用聊天端设计系统；
- 信息密度更高，但不回到传统 admin 样式；
- 页面组织优先为：标题 + 简介 + 分段配置组 + 列表/表格/表单模块 + 辅助说明。

#### 重点避免

- 大面积灰底 + 白卡片宫格；
- 配置页面被无意义边框切得过碎；
- 聊天端现代，管理端却回退成旧式后台。

## 13. 移动端设计策略

### 13.1 总体定位

移动端不是桌面缩放版，而是 **原生 app 感的聚焦型工作台**。

目标是：

- 更轻；
- 更稳；
- 更聚焦；
- 单任务优先。

### 13.2 通用原则

1. **单主区域优先**：同一时刻只强调一个主任务区；
2. **输入体验优先**：键盘、输入框、生长高度、发送动作要稳定；
3. **导航弱化但可达**：侧滑抽屉、sheet、二级页替代常驻多栏；
4. **减少视觉碎片**：避免大量小卡片、小 badge、小工具条同时出现。

### 13.3 聊天页移动端

#### 确认方案

- **原生 app 感**；
- **顶部导航 + 侧滑会话抽屉**。

#### 默认主视图

- 顶部：轻量 header；
- 中部：消息流；
- 底部：固定 composer。

#### 次级能力承载

- 会话列表：抽屉式；
- 模型/agent/状态：顶部轻入口或 bottom sheet；
- 详情与上下文：二级面板或独立页。

#### 设计要求

- 消息流具备舒适边距，不贴边；
- 代码块、引用、列表在窄屏下保持可读；
- composer 自适应高度、键盘弹起后布局稳定；
- 辅助入口折叠收纳，避免底部工具条拥挤。

### 13.4 首页移动端

采用纵向叙事结构：

1. 价值主张；
2. 产品界面片段；
3. 核心能力说明；
4. 协作能力；
5. CTA。

要求每一屏只有一个主重点，避免桌面 feature 区简单堆成手机长页面模板。

### 13.5 登录页移动端

- 更轻、更稳；
- 不做厚重居中卡片；
- 重点保证输入尺寸、触控区域、错误与加载反馈清晰。

### 13.6 管理端移动端

手机端不追求完全等价于桌面端：

- 保留查看、搜索、少量关键操作与轻量编辑；
- 复杂配置进入分步表单、独立详情页、分段编辑页；
- 避免密集表格与多列矩阵在手机上硬排。

---

## 14. 交互与动效规范

### 14.1 动效原则

- 仅做微交互；
- 时长以 150-220ms 为主；
- 进入采用 ease-out，退出采用 ease-in；
- 不做持续装饰动画；
- 遵守 `prefers-reduced-motion`。

### 14.2 Hover / Focus / Active

- hover：颜色 / 边框 / 背景轻变化；
- focus：必须清晰可见，尤其键盘导航；
- active：轻微压感，不产生位移跳动。

### 14.3 Loading / Empty / Error

- 优先局部 loading / skeleton；
- 避免整个区域突然空白；
- 错误提示需具有清晰语义与可恢复动作；
- 流式输出状态应沉着稳定，而不是躁动闪烁。

---

## 15. 跨页面统一要求

首页、登录页、聊天页、管理端必须共享以下统一特征：

- 同一套色彩明度逻辑；
- 同一套字体层级；
- 同一套按钮 / 输入 / 标签 / 状态规则；
- 同一套间距节奏；
- 同一套 hover / focus / loading / empty state 语言。

最终效果应让用户感受到：

> 这是一个完整产品，而不是几个页面拼起来。

## 17. 验收标准

本次重构完成后，应至少满足以下结果：

### 17.1 视觉与品牌一致性

- 首页、登录页、聊天页、管理端明显属于同一产品；
- 深浅主题都自然成立；
- 不再存在明显 AI 模板页 / 传统后台模板痕迹。

### 17.2 聊天体验

- 消息区成为视觉中心；
- 输入区在桌面端与移动端都稳定、好用；
- 侧栏与状态区不再压过主线；
- 长时间阅读和输入的舒适性显著提升。

### 17.3 管理端体验

- 页面更清晰、更统一；
- 信息密度提升但不沉重；
- 表格、表单、列表、状态展示拥有一致语言。

### 17.4 移动端体验

- 聊天页具备原生 app 感；
- 顶部导航 + 侧滑抽屉方案成立；
- 输入、阅读、切会话体验稳定；
- 管理端移动端具备基本可用与清晰层级。

### 17.5 设计系统可维护性

- 共享 token 与基础组件可复用；
- 新页面不会再轻易偏离整套系统；
- 样式和组件边界对后续演进友好。

## 18. 后续实施建议

后续实施阶段建议按以下顺序推进：

1. **先做设计 token 与 shared UI 基础重构**；
2. **再做聊天页骨架重构**（最高优先级）；
3. **同步统一首页 / 登录页气质**；
4. **最后做管理端高密度化统一**；
5. 在每一阶段都同时验证：
   - 浅色主题；
   - 深色主题；
   - 移动端；
   - 键盘/焦点/状态反馈。

## 19. 结论

本次重构不是单点页面美化，而是对当前前端进行一次 **统一设计系统 + 页面结构 + 跨端体验** 的整体升级。

最终要实现的不是“更像 AI 产品”，而是：

> **更像一个被认真打磨过、适合开发者和小团队长期使用的现代工作台。**
